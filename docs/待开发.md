[ ] 中转网关 SSE tool_calls 兼容性兜底
    - 现象：apinova.cc 中转 GPT-5.5 时，第 2 轮请求返回 400
      `Missing required parameter: 'input[3].name'`（Responses API 路径）
    - 根因：apinova 内部把 Chat Completions 转 Responses API；其 SSE 流
      没透传 `delta.tool_calls[i].function.name`，导致 acecode 收到
      tool_calls 但 function_name 为空（日志：`WRN Unknown tool: `），
      acecode 把空 name 的 tool_call 写回 messages_，下一轮回传被 400
    - 复现日志：N:\Users\shao\dbcoding4\acecode.log
    - 兜底方案：openai_provider.cpp 的 SSE flush 处（约 270-284、382-397
      两处）发现 `tc.name.empty()` 时丢弃该 tool_call 并 LOG_ERROR；
      若一轮内全部被丢弃，按 text-only 结束 loop，避免毒数据进历史
[ ] YOLO / 无人值守场景下的 AskUserQuestion 打断策略
    - 背景：当前 `--yolo` / `PermissionMode::Yolo` 只跳过工具权限确认，
      不会抑制 `AskUserQuestion`。模型一旦调用该工具，TUI 会弹 overlay，
      daemon/web 会发 `question_request` 并阻塞等待用户回答。
    - 问题：用户已经选择 YOLO 时，预期通常是“别再中断我，自己推进”；
      后续定时任务、headless CLI、后台 daemon、远程/服务模式、子代理等
      无人值守场景也不能依赖实时人工回答。
    - 已知参考：
      - claudecodehaha: `AskUserQuestionTool.requiresUserInteraction()` 会让
        bypassPermissions 仍然保留交互工具提示；但它还有 `dontAsk` 模式，
        可把 ask 转成 deny，避免无人值守时卡住。
      - Codex: `request_user_input` 默认只在 Plan mode 可用，Default mode
        会拒绝该工具；Full Access 只表示权限/沙箱放开，不等于允许提问。
    - 待研究方向：
      - 是否新增独立配置，例如 `question_policy = ask | deny | timeout`，
        不把“权限全放开”和“禁止业务提问”混在同一个 YOLO 语义里。
      - YOLO 默认是否映射为 `deny`：`AskUserQuestion` 直接返回可恢复错误，
        提醒模型“当前无人值守/禁止打断，请自行做合理假设继续”。
      - daemon 切换到 YOLO 时，除现有 `AsyncPrompter::resolve_all(Allow)`
        之外，是否也应关闭/取消未决 `AskUserQuestion` 请求。
      - 是否按运行入口区分默认值：TUI/desktop active session = ask，
        background daemon / automation / headless CLI / subagent = deny 或 timeout。
      - 若采用 timeout，超时后的工具输出、前端 pending question 清理、
        最终回复里的“采用了什么假设”都要有明确契约。
